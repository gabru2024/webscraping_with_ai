Common Base Dockerfile Structure

A well-structured Dockerfile follows a logical order that optimizes build caching and image size. Here's what a typical base Dockerfile contains:

1. Base Image

The foundation of your Docker image. Choose based on your application needs:

FROM python:3.11-slim
FROM node:18-alpine
FROM ubuntu:22.04
FROM golang:1.21
üí° Tip: Use slim or alpine variants for smaller image sizes. Alpine Linux images are typically much smaller but may require additional dependencies.

2. Metadata Labels

LABEL maintainer="your-email@example.com" LABEL version="1.0" LABEL description="Application description"

Labels help organize and document your images. They're metadata that doesn't affect functionality but is useful for management.

3. Working Directory

WORKDIR /app

Sets the working directory for all subsequent commands. Creates the directory if it doesn't exist.

4. Environment Variables

# Python example ENV PYTHONUNBUFFERED=1 ENV PYTHONDONTWRITEBYTECODE=1 # Node.js example ENV NODE_ENV=production

Set environment variables that will be available to your application at runtime.

5. System Dependencies

RUN apt-get update && apt-get install -y \ build-essential \ curl \ git \ && rm -rf /var/lib/apt/lists/*
‚ö†Ô∏è Important: Always clean up apt cache with rm -rf /var/lib/apt/lists/* to reduce image size. Combine RUN commands to reduce layers.

6. Application Dependencies

Python Example:

COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt

Node.js Example:

COPY package*.json . RUN npm ci --only=production
üí° Best Practice: Copy dependency files first, then install. This allows Docker to cache the dependency layer and only rebuild when dependencies change.

7. Application Code

COPY . .

Copy your application code into the image. This is typically done after dependencies to maximize cache efficiency.

8. Expose Ports

EXPOSE 8000 # or multiple ports EXPOSE 8000 8080

Documents which ports your application listens on. This is informational and doesn't actually publish the ports.

9. Health Check (Optional)

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \ CMD curl -f http://localhost:8000/health || exit 1

Allows Docker to check if your container is still working. Useful for orchestration tools like Kubernetes.

10. Entry Point / Command

# Option 1: CMD CMD ["python", "app.py"] # Option 2: ENTRYPOINT + CMD ENTRYPOINT ["python"] CMD ["app.py"]
üìö Difference: CMD can be overridden at runtime, while ENTRYPOINT is fixed. Use ENTRYPOINT when you want a fixed command with variable arguments.

Complete Example

# Python Flask Application Example FROM python:3.11-slim LABEL maintainer="dev@example.com" LABEL version="1.0" WORKDIR /app ENV PYTHONUNBUFFERED=1 ENV PYTHONDONTWRITEBYTECODE=1 RUN apt-get update && apt-get install -y \ curl \ && rm -rf /var/lib/apt/lists/* COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 8000 HEALTHCHECK --interval=30s --timeout=3s \ CMD curl -f http://localhost:8000/health || exit 1 CMD ["python", "app.py"]

Best Practices

üéØ Layer Ordering

Order commands from least to most frequently changing. Dependencies change less often than code.

üì¶ Multi-stage Builds

Use multi-stage builds to keep final images small by copying only necessary artifacts.

üîí Security

Never include secrets in Dockerfiles. Use environment variables or secrets management.

üßπ Clean Up

Remove package manager caches and temporary files in the same RUN command to reduce size.

üë§ Non-root User

Create and use a non-root user for running applications to improve security.

üìù .dockerignore

Use .dockerignore to exclude unnecessary files from the build context, speeding up builds.
